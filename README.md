# constellation-stream: Heliosnet streaming data plane

[![CI](https://github.com/maisymylod/constellation-stream/actions/workflows/ci.yml/badge.svg)](https://github.com/maisymylod/constellation-stream/actions/workflows/ci.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)

The **production-scale streaming evolution** of the
[`constellation`](https://github.com/maisymylod/constellation) data plane. Where
`constellation` batches a Kafka topic into TimescaleDB, this repo turns the same
Heliosnet telemetry into a stateful, event-time stream-processing pipeline:

**Kafka/Redpanda -> stream processor -> Apache Iceberg lakehouse -> Grafana**,
with windowed anomaly aggregation, watermarks and late-data handling, and an
exactly-once (idempotent) sink.

Every number below is produced by running the code. See
[What is real vs simulated](#what-is-real-vs-simulated) for exactly what scale
was run.

## Architecture

```
 telemetry (simulated, labelled, event-time, with injected late records)
        |
   Kafka / Redpanda  (topic keyed by sat_id)
        |
   Stream processor  (stateful, event-time)
     - 30s tumbling windows keyed by sat_id
     - bounded-out-of-orderness watermark (2s)
     - allowed lateness (10s): late records UPDATE a fired window (new revision)
     - too-late records -> dead-letter (never dropped)
     - windowed anomaly classification: nominal / drift / dropout / thermal_runaway
     Two runtimes, identical semantics:
       (a) flink/anomaly_job.py     real Apache Flink cluster
       (b) src/.../processor.py     cluster-free reference engine (tests, CI, bench)
        |
   Exactly-once / idempotent sink  (upsert by (sat_id, window_start)+revision;
                                    committed-offset fence for replays)
        |
   Apache Iceberg lakehouse  (partitioned by sat_id; filesystem or MinIO/S3)
        |
   Grafana  (provisioned dashboards: anomaly counts, late-data, drilldown)
```

Full design in [docs/architecture.md](docs/architecture.md).

## Quickstart

No infrastructure needed for the headline. The whole streaming pipeline (windows,
late-data, exactly-once) runs in-process:

```bash
make install        # core + dev (numpy, pytest); no JVM, no cluster
make test           # unit tests + quality gate (windowing, late-data, exactly-once)
make bench          # REAL benchmark -> artifacts/bench.json + artifacts/BENCH.md
```

Land window results into a real Apache Iceberg table and serve Grafana:

```bash
make install-lake   # adds pyiceberg + pyarrow
make pipeline       # generate -> window -> idempotent sink -> Iceberg table
make demo           # Redpanda + MinIO + Grafana, export dashboard data
```

Then open:

- **Grafana**: http://localhost:3000 (heliosnet / heliosnet)
- **MinIO console**: http://localhost:9001 (heliosnet / heliosnet123)

Run the real Flink cluster and submit the PyFlink job:

```bash
make flink-up       # Flink jobmanager + taskmanager (Flink UI on :8081)
make flink-submit   # submit flink/anomaly_job.py
```

## Real results

Produced by `python -m constellation_stream.bench --rows 5000000 --query-iters 100`
(deterministic, seed 42) on the build machine. Regenerated by CI at a smaller
scale on every push.

**Scale actually run: 5,000,000 telemetry records through the full
stateful pipeline (windowing + watermark + late-data + idempotent sink), on a
single core.**

| Metric | Value |
|---|---|
| Records processed | **5,000,000** |
| Ingest / processing throughput | **38,819 records/s** |
| Window results produced | 166,800 |
| Late records folded into windows | 13,668 |
| Query p99 latency (top anomaly per satellite) | **32.968 ms** |
| Query backend | in-memory window results |

Hardware: macOS-26.2-arm64-arm-64bit-Mach-O, 10 CPUs, Python 3.14.2.

Reproduce: `python -m constellation_stream.bench --rows 5000000 --query-iters 100`
(or `make bench-big`). The committed `artifacts/BENCH.md` is the verbatim output
of the run above.

### Reaching billions of events

The number above is the largest scale that completed in this single-machine,
single-core build environment via the cluster-free reference engine. The pipeline
keys by `sat_id` with no cross-key state, so it scales out horizontally with zero
logic changes. To reach billions:

```bash
# 1. Start the real Flink cluster and a partitioned Kafka topic.
make flink-up
# 2. Run the PyFlink runtime (flink/anomaly_job.py) at parallelism P across a
#    multi-node TaskManager pool, landing into Iceberg on MinIO/S3.
make flink-submit            # FLINK_PARALLELISM and broker/partition count drive scale
```

At parallelism `P` sustaining the per-core rate measured above, wall time for `N`
events is about `N / (P x rate)`. The per-core rate is what this repo reproduces;
the rest is horizontal scale-out on a multi-node cluster, which a single-machine
build cannot bundle. This is stated plainly rather than claimed as a run that did
not happen. See [docs/architecture.md](docs/architecture.md#the-path-to-billions).

## What is real vs simulated

- **Simulated:** the telemetry. There is no physical constellation;
  `generator.py` synthesises labelled telemetry over the Heliosnet channel schema
  and deliberately injects out-of-order / late records so the late-data path runs
  on real disorder, not a mock.
- **Real (runs here, exercised by CI):** the stream-processing semantics
  (event-time tumbling windows, watermarks, allowed-lateness updates,
  dead-lettering), the exactly-once / idempotent sink, the Apache Iceberg write
  (and its idempotent re-run), and every benchmark number.
- **Provided and runnable, but not the headline:** the real Apache Flink job
  (`flink/anomaly_job.py`) on the committed Flink Docker cluster. The build
  environment runs Python 3.14 (no PyFlink wheel) and Java 11 (Spark 4 needs 17+),
  so the **in-environment headline uses a purpose-built cluster-free reference
  engine** with the same semantics, on a single core. The Flink job is the same
  logic at cluster fidelity, for the scale-out path above.

Trust boundaries and limitations: [docs/assumptions.md](docs/assumptions.md).

## Quality gate (CI)

[`.github/workflows/ci.yml`](.github/workflows/ci.yml) runs three jobs; the build
fails on regression:

- **Tests**: full unit suite + a coverage floor (>= 85%). The windowing,
  late-data, and exactly-once logic is unit-tested deterministically with no
  cluster.
- **Pipeline**: runs the local pipeline into an Apache Iceberg table, runs it a
  second time and asserts the table is unchanged (idempotency), then runs a real
  small-scale benchmark.
- **Flink job**: byte-compiles the PyFlink job so it cannot rot.

A weekly [`reproduce.yml`](.github/workflows/reproduce.yml) re-derives the
CI-scale benchmark from a clean checkout under the `heliosnet-ci[bot]` identity
and commits the refreshed artifacts only if they genuinely changed.

## Project layout

```
src/constellation_stream/
  schema.py       vendored Heliosnet channels + anomaly taxonomy + window classifier
  config.py       seeded generation + windowing/watermark config
  generator.py    deterministic event-time telemetry with injected lateness
  processor.py    cluster-free stateful event-time stream engine (tests/CI/bench)
  sink.py         exactly-once / idempotent sink (upsert + offset fence)
  lake.py         Apache Iceberg schema + catalog/table (pyiceberg)
  pipeline.py     end-to-end local pipeline: generate -> window -> sink -> Iceberg
  bench.py        real throughput + p99 benchmark harness
  export.py       window results -> grafana/data/windows.json
  produce.py      Kafka/Redpanda producer (cluster path)
flink/anomaly_job.py   real PyFlink job (same semantics, cluster fidelity)
grafana/               provisioned datasource + dashboards (committed JSON)
docker-compose.yml     Redpanda + MinIO + Grafana (+ Flink profile)
tests/                 windowing, late-data, exactly-once, generator, schema
docs/                  architecture + assumptions/honesty
artifacts/             committed real benchmark result (bench.json + BENCH.md)
```

## Part of a larger system

`constellation-stream` is the streaming data plane of the Heliosnet ground-system
suite. It reuses the telemetry schema and anomaly taxonomy of the
[`constellation`](https://github.com/maisymylod/constellation) data plane; the
[`groundstation`](https://github.com/maisymylod/groundstation) agent graph and
[`groundstation-train`](https://github.com/maisymylod/groundstation-train)
reasoning core consume anomalies of the same taxonomy. Each project stands alone
and runs on its own.
